# Grill — Adversarial Review

以攻击者和极端场景视角审视当前变更。与 `/code-review`（合规检查）不同，本命令主动尝试"打破"代码。

1. 获取变更文件: `git diff --name-only HEAD`
2. 阅读每个变更文件的完整 diff 和上下文

3. 从以下维度逐一挑战每个变更：

**边界 & 极端输入:**
- 空值、null、undefined、NaN 路径是否安全？
- 超大输入（10MB+ 字符串、百万条记录）会怎样？
- 恶意构造的输入（注入、路径遍历、特殊字符编码）能否穿透？
- 并发/竞态：两个调用者同时操作同一资源会怎样？

**故障模式:**
- 网络断开、超时、部分响应会触发什么行为？
- 磁盘满、文件锁、权限不足时会静默失败还是崩溃？
- 依赖服务返回非预期数据（空 JSON、HTML 错误页、超长响应）？

**设计假设挑战:**
- 这段代码依赖了什么隐式假设？在什么条件下假设会失效？
- 数据量增长 100 倍后性能和内存会怎样？
- 如果其他团队/模块不按预期方式调用这个接口？

4. 为每个发现生成：
   - **场景描述**: 具体的破坏路径，不要泛泛而谈
   - **严重度**: CRITICAL / HIGH / MEDIUM
   - **当前防御**: 代码现在如何应对（或完全没有防御）
   - **建议**: 具体的防御措施或测试用例

5. CRITICAL 级别的挑战必须解决后才建议创建 PR

每个挑战必须引用具体的文件:行号和代码片段。禁止笼统的"建议加强输入验证"之类的废话。
