# 手动命令使用说明

## 目标

把手动命令从“知道名字但不知道何时用”变成“有明确入口、顺序和预期结果”的操作手册。

## 使用原则

1. 手动命令用于高价值但不适合默认自动运行的流程。
2. 命令应在用户或任务上下文明确需要时才调用。
3. 命令说明必须包含“适用场景”和“不要乱用的场景”。

## 命令说明

### `/plan`

何时使用：

- 需求已经清楚，但实现步骤较多
- 任务跨多个模块
- 需要把工作拆给其他 agent 或后续会话

不要在这些场景滥用：

- 单文件小修
- 单次问答
- 不到 30 分钟即可完成的小任务

预期结果：

- 产出一份结构化实施计划
- 标明任务分块、依赖、验证方法

### `/tdd`

何时使用：

- bugfix 且需要防回归
- 核心逻辑变更
- 新功能有明确可测行为

不要在这些场景滥用：

- 纯文档工作
- 样式微调
- 不具备合理测试边界的一次性脚本

预期结果：

- 先写失败测试
- 最小实现
- 再验证通过

### `/verify`

何时使用：

- 准备宣称“完成”
- 需要做合并前检查
- 涉及接口、配置、部署或行为变更

预期结果：

- 汇总验证命令
- 给出通过 / 失败状态
- 明确剩余风险

### `/code-review`

何时使用：

- 共享模块改动
- 高风险逻辑
- 准备交付或合并

预期结果：

- 找问题优先于写总结
- 输出按严重级别排序的发现
- 附带残留风险说明

### `/save-session`

何时使用：

- 工作中断
- 需要跨会话续做
- 需要把阶段结论显式沉淀

预期结果：

- 保存当前目标、进度、阻塞项、下一步动作

### `/resume-session`

何时使用：

- 恢复跨会话工作
- 需要快速接上次任务上下文

预期结果：

- 恢复最近 session 摘要
- 明确上次停在哪、接下来做什么

## 推荐调用顺序

### 轻量开发

`/verify`

### 标准开发

`/plan` -> 实施 -> `/verify`

### 高风险开发

`/plan` -> `/tdd` -> 实施 -> `/code-review` -> `/verify`

### 多 agent / 长任务

`/plan` -> 阶段性 `/save-session` -> `/resume-session` -> `/verify`

## 命令说明模板

每个命令文档统一写法：

```text
用途
适用场景
不适用场景
输入格式
最小示例
预期输出
与其他命令的关系
失败时如何处理
```

## 文档层面的硬要求

1. 所有命令必须写出“不适用场景”。
2. 所有命令必须写出“预期输出”。
3. 所有命令必须写出“与其他命令的衔接关系”。
4. 任何命令如果依赖 Heavy 模式，文档必须明确标注。
