# 权限模型

## 目标

把“自治效率”和“操作安全”分开处理，避免为了少点确认就把危险能力默认放开。

## 三层权限

### Layer 1：默认允许

这些命令可作为低风险日常能力：

- 只读文件系统命令
- `git status`
- `git diff`
- `git log`
- 局部测试
- lint
- typecheck
- 只读搜索与静态分析

要求：

- 不写远程
- 不破坏文件
- 不改变部署或数据状态

### Layer 2：条件允许

这些命令需要任务上下文允许，但不一定每次人工确认：

- `git add`
- `git commit`
- 创建分支
- 本地构建
- 依赖安装
- 本地生成产物

条件：

- 当前任务明确需要
- 结果可回滚
- 不涉及远程副作用

### Layer 3：必须人工确认

这些动作永远不走默认自治：

- `git push`
- `deploy`
- `publish`
- 数据迁移执行
- 远程写操作
- 大规模删除
- 修改生产环境资源
- 涉及 secrets 的真实操作

## 禁止的大前缀白名单

默认禁止以下宽泛授权方式：

- `Bash(node *)`
- `Bash(npx *)`
- `Bash(git *)`
- `Bash(python *)`
- `Bash(sh *)`

原因：

- 范围过大
- 很难审计
- 容易把局部问题升级成系统性风险

## 建议批准策略

### 按任务类别批准

- 测试类
- 构建类
- 只读 Git 类
- 本地生成类

### 不按解释器批准

不要因为“都是 node 命令”就整体批准。

## 权限与模式联动

### Fast

- 只使用 Layer 1

### Standard

- 默认 Layer 1
- 需要时申请 Layer 2

### Heavy

- Layer 1 + Layer 2
- 任何 Layer 3 都必须人工确认

## 权限审计要求

每次高风险批准都要记录：

- 请求命令
- 请求原因
- 当前模式
- 影响范围
- 是否可回滚

## 推荐配置思路

1. 把“允许哪些事”按动作分类，而不是按解释器分类。
2. 只对白名单测试命令和只读命令做长期批准。
3. 对远程写、发布、删除、迁移永远保留人工闸门。
