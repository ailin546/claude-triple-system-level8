# 降级与恢复策略

## 目标

确保系统任一组件失效时，仍能继续工作，而不是整套流程卡死。

## 总原则

1. 优先保住核心工作流。
2. 失败的自动化组件可以停，任务不能停。
3. 降级必须显式，不允许静默失效。
4. 重型能力失败时，必须退回单 agent 安全模式。

## 场景一：SessionStart 异常

表现：

- 无法读取最近会话
- 包管理器识别失败
- memory 文件损坏

处理：

1. 跳过历史恢复
2. 只保留当前会话
3. 输出 warning
4. 继续进入 task routing

## 场景二：轻量 hook 失败

表现：

- 格式化失败
- 局部检查超时
- 风险扫描脚本报错

处理：

1. 记录 warning
2. 不阻塞当前任务
3. 标记该 hook 为临时降级

## 场景三：shared-state 异常

表现：

- board 写入失败
- 冲突无法解决
- stale worker 清理异常

处理：

1. 停止 shared-state-sync
2. 停止多 agent 新任务派发
3. 输出“已降级为单 agent 模式”
4. 把最近 handoff 汇总到本地 session

## 场景四：memory 异常

表现：

- sprint memory 写入失败
- session 摘要损坏
- 长期记忆冲突

处理：

1. 暂停 memory 写入
2. 不影响当前任务继续
3. 仅在结束时输出“本轮未持久化”

## 场景五：权限异常

表现：

- 自动批准规则失效
- 本地 override 不兼容
- 本该确认的高风险命令被误分类

处理：

1. 立即收缩到最保守权限集
2. 后续高风险动作全部要求人工确认
3. 记录权限异常事件

## 场景六：验证链路异常

表现：

- 测试命令超时
- 构建工具不存在
- 验证脚本与项目不兼容

处理：

1. 汇报无法完成的验证项
2. 保留已完成的验证证据
3. 禁止声称“全部通过”

## 统一降级状态

建议所有组件都输出统一状态：

- `ok`
- `warn`
- `degraded`
- `blocked`

说明：

- `ok`：正常运行
- `warn`：局部失败，但主流程可继续
- `degraded`：已关闭部分子系统，主流程可继续
- `blocked`：必须人工处理

## 恢复条件

只有在以下条件满足时，才允许从降级恢复：

1. 失败根因已明确
2. 再次运行通过基本自检
3. 不会破坏当前任务上下文

## 运行要求

重构后必须把“降级”当作一等能力，而不是异常兜底。
